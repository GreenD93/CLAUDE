# 글로벌 작업 선호도

## 사용자 프로그래밍 스타일

> 관찰된 의사결정 패턴. 모든 프로젝트에서 이 스타일에 맞춘다.

- **구조적 해결을 선호한다.** 증상 패치(하드코딩 분기, 플래그)를 제시하면 바로 교정한다.
- **"왜?"를 먼저 묻는다.** 기능을 넣기 전에 "이게 지금 필요한 증거가 있나?"를 확인한다.
- **실행 후 판단을 바꿀 수 있다.** 초기 결정도 돌려보고 문제가 드러나면 전환한다.
- **기존 검증된 패턴을 먼저 참고한다.** "이미 동작하는 유사 사례가 있나?"부터 확인.
- **패턴이 발견되면 핵심 철학으로 승격한다.** 체크리스트 항목이 아닌 설계 원칙으로 격상.
- **네이밍에 집요하다.** 역할이 안 드러나는 이름을 바로 지적한다.
- **더미 데이터를 허용하지 않는다.** 실제 데이터를 조사해서 채운다.
- **문서 동기화를 빼먹으면 지적한다.** 코드 변경 = 문서 변경. 둘은 하나의 작업이다.

## 커뮤니케이션
- 한국어로 대화한다
- 불필요한 설명 없이 간결하게
- 코드 변경 전 의도가 불명확하면 먼저 확인

## 설계 철학

> "동작하는 코드를 만든 다음, 다음 사람이 읽고 확장할 수 있을 때까지 구조를 다듬는다."

### 핵심 원칙

- **구조적 안전망 > 정확도 의존**: 외부 의존(LLM, API, 모델)이 실패해도 사용자 경험이 보장되는 코드 구조를 만든다. 한 컴포넌트의 정확도에만 의존하지 않는다. 실패 경로에 graceful fallback을 항상 설계한다.
- **코드로 강제 > 문서로 강제**: 규칙이 코드 구조에 의해 강제되면 위반이 불가능하다. 문서에만 의존하면 반드시 누군가 위반한다. 책임 경계를 인터페이스·계층으로 강제한다.
- **레퍼런스 구현체 기반 설계**: 추상 문서 대신 동작하는 레퍼런스를 만든다. 새 기능은 검증된 구현체를 복사해서 시작. 템플릿은 일관된 네이밍·구조를 유지해야 복사 후 즉시 동작.
- **반복적 정련 (Iterative Refinement)**: 기능 개발은 빠르게, 그 후 구조·네이밍·문서를 계속 정제. 한 번에 완벽하게가 아니라 점진적으로 다듬는다.

### 구조 원칙

- **단일 소유권 (SSOT)**: 하나의 정보는 한 곳에서만 정의. 중복은 불일치의 씨앗.
- **역할 분리**: 한 모듈 = 한 역할. 공통 동작은 추상 클래스로, 개별 구현은 상속+오버라이드.
- **선언으로 조립**: 설정 파일(yaml/json)로 선언하면 프레임워크가 자동 조립. 개발자는 선언만, 배선은 코드가.
  - **필드 생략 ≠ 빈 값**: 생략은 "모름/미정", 빈 값(`[]`, `{}`)은 "의도적으로 사용 안 함".
- **레이어 간 경계 엄수**: 레이어 간 통신은 공개 인터페이스(이벤트, 주입 파라미터)로만. 내부 저장소 직접 참조 금지.
- **Core에 비즈니스 로직 금지**: Core는 인터페이스와 공통 패턴만. 비즈니스 로직은 프로젝트 계층에서.
- **모든 루프에 상한선**: 무한루프·무한대기 없음. 상한 초과 시 자연스럽게 복구하는 탈출구.
- **정책값은 설정 가능하게**: 매직넘버는 .env/config로 분리. 코드 수정 없이 정책 변경 가능하게.

### UX·Hybrid 원칙

- **UX에서 구조를 도출**: "사용자가 어떤 경험을 해야 하나?"에서 출발하여 데이터·이벤트 설계를 역추적. 데이터 양·성격에 맞는 UI를 먼저 결정하고 API를 설계.
- **Hybrid 감지 패턴**: 코드(비용 0, 결정론적) → LLM 폴백(범용, 확률적). 하드코딩으로 모든 케이스를 대응하지 않는다. 키워드로 커버 가능하면 LLM 호출 없이, 키워드 누락·자연어 변형은 LLM이 판단.
- **정상 경로와 엣지케이스를 동시에 설계**: "결과 없음", "대응 불가", "타임아웃" 등의 사용자 경험을 정상 경로와 함께 설계한다. 엣지케이스가 결정론적이면 코드로 처리.

## 코딩 원칙
- 오버엔지니어링 금지: 현재 필요한 것만 구현. 아직 돌려보지 않은 최적화는 하지 않는다.
- 새 파일 생성보다 기존 파일 편집 우선
- 주석은 로직이 자명하지 않을 때만
- 사용하지 않는 코드·주석 남기지 않기

## 구조·가독성
- **네이밍 = 역할 설명**: 파일명·클래스명·설정 경로 모두 이름만 보고 역할을 알 수 있어야 한다. 범용 이름 금지.
- **일관된 네이밍**: 같은 역할에는 전 서비스에서 같은 이름. 템플릿이 불일치하면 복사한 프로젝트도 불일치.
- **파일 하이어러키 준수**: 한 파일에 역할이 다른 클래스 3개 이상이면 분리. flat 구조 금지.
- **전체 구조가 한눈에 읽혀야 한다**: 개별 코드보다 아키텍처 전체 그림이 먼저 보이게.
- **워크플로우 흐름은 코드에서 직접 읽혀야 한다**: 추상화 뒤에 프로세스 흐름이 숨지 않아야 한다.
- **문서 ↔ 코드 동기화**: 코드 변경 시 관련 문서 즉시 반영. 문서는 코드 변경의 일부.
- **인터페이스가 코드 구조에 드러나야 한다**: 서비스별 UI 계약, API 엔드포인트 등이 폴더/파일 구조에서 바로 보여야 확장성이 생긴다.

## 수정의 깊이 — 3단계를 항상 완주한다

```
  ① 원인 분석    증상이 아닌 근본 원인을 찾는다.
  ② 적절한 계층에서 수정
                  개별 프로젝트 문제 → 프로젝트에서 수정
                  여러 프로젝트에서 반복될 패턴 → Core로 승격
  ③ 문서화 + 재발 방지
                  패턴 기록, CHANGELOG, 테스트 추가
```

## 버그 수정 시 동일 패턴 전수 점검

```
  ① 발생 지점 수정
  ② grep으로 동일 패턴을 사용하는 모든 파일을 전수 조사
  ③ 같은 문제가 있는 곳을 모두 수정
  ④ 규칙으로만 강제되고 코드로 강제되지 않은 것이면 코드 구조를 바꾼다
```

## 래퍼·어댑터 추가 시 전수 마이그레이션

```
  ① grep으로 이전 메서드의 모든 호출처를 전수 조사
  ② 호출처를 전부 새 래퍼로 교체
  ③ "유일한 진입점" 명시. 이전 경로를 남기지 않는다.
```

## 프레임워크 위에서 작업할 때

```
  ① 프레임워크 API로 목표 달성 가능한지 먼저 확인
  ② 불가능하면 구조적 대안 검토 (패치 아닌)
  ③ 같은 방향의 패치를 3번 이상 반복하면 방향 자체가 틀린 것
```

## 문서 작성 스타일
- 실제 입력 예시로 시작 → 큰 그림 → 세부 → 관계도
- 역방향 매핑 테이블 ("뭘 바꾸려면 어디를 보나")
- ASCII 다이어그램 (차트만 보고 구조를 파악할 수 있는 수준)
- 프로세스 순서: `① ② ③`

## 작업 방식
- 큰 변경은 계획 먼저 제시 후 진행
- 기능 추가 후 기존 테스트 통과 확인
- 같은 실수 2번 이상 → "흔한 실수" 테이블에 추가
- 세션이 길어지면 커밋 후 다음 세션으로 넘긴다

## 문서 계층

```
~/.claude/CLAUDE.md                  ← [이 파일] 글로벌. 모든 프로젝트에 적용.
프로젝트/CLAUDE.md                   ← 프로젝트별. git에 포함 (팀 공유).
프로젝트/STRUCTURE.md                ← ASCII 구조도 전용 (CLAUDE.md에서 링크).
~/.claude/projects/{경로}/memory/    ← 프로젝트별 실무 지식 (로컬, 본인만).
```
