# 글로벌 작업 선호도

## 커뮤니케이션
- 한국어로 대화한다
- 불필요한 설명 없이 간결하게
- 코드 변경 전 의도가 불명확하면 먼저 확인

## 코딩 원칙
- 오버엔지니어링 금지: 현재 필요한 것만 구현
- 새 파일 생성보다 기존 파일 편집 우선
- 주석은 로직이 자명하지 않을 때만
- 사용하지 않는 코드·주석 남기지 않기

## 설계 철학

> "동작하는 코드를 만든 다음, 다음 사람이 읽고 확장할 수 있을 때까지 구조를 다듬는다."
> 기능 개발은 빠르게, 그 후 구조·네이밍·문서를 계속 정제하는 반복적 정련(iterative refinement).

- **단일 소유권 (SSOT)**: 하나의 정보는 한 곳에서만 정의. 중복은 불일치의 씨앗이다.
- **역할 분리**: 각 클래스가 자기 역할만 하고, 책임 경계를 코드 구조로 강제한다. 규칙 문서에만 의존하지 않는다.
- **선언으로 조립**: 설정 파일(yaml/json)로 선언하면 프레임워크가 자동 조립. 개발자는 선언만, 배선은 코드가.
- **모든 루프에 상한선**: 무한루프·무한대기 없음. 상한 초과 시 자연스럽게 복구하는 탈출구를 항상 마련한다.
- **Core에 비즈니스 로직 금지**: Core는 인터페이스와 공통 패턴만. 비즈니스 로직은 프로젝트 계층에서.

## 구조·가독성 원칙
- **네이밍 = 역할 설명**: 파일명·클래스명·설정 경로 모두 이름만 보고 역할을 알 수 있어야 한다
- **설정 파일의 경로·값은 처음 보는 사람이 바로 이해할 수 있게** 주석이나 규칙을 명시
- **하드코딩보다 설정 가능한 구조**: 매직넘버·정책값은 설정 파일로 분리, 프로젝트별 커스텀 가능하게
- **전체 구조가 한눈에 읽혀야 한다**: 개별 코드보다 아키텍처 전체 그림이 먼저 보이게
- **문서 ↔ 코드 동기화**: 코드 변경 시 관련 문서 즉시 반영
- **예외 상황에서 자연스러운 복구**: 에러·엣지케이스에서 사용자 경험이 끊기지 않게

## 문서 작성 스타일

### 구조
- **실제 입력 예시로 시작**: 추상 설명 전에 `"엄마한테 1만원 보내줘"` 같은 구체적 시나리오부터
- **큰 그림 → 세부 → 관계도 순서**: 전체 흐름 먼저, 각 클래스 상세, 마지막에 클래스 간 관계
- **각 클래스는 3단 구조**: 역할 / 소유(의존성) / 핵심 메서드
- **역방향 매핑 테이블**: "뭘 바꾸려면 어디를 보나" — 목적 → 수정 파일
- **대비 테이블**: 코드 vs LLM, 실수 vs 올바른 방법 등 양쪽을 나란히
- 프로젝트의 실행 흐름·클래스 관계·데이터 흐름은 ASCII 다이어그램으로 문서화
- **차트만 보고 구조를 파악할 수 있어야 한다** — 본문 없이도 이해 가능한 수준
- 박스 `┌─ ClassName ─┐`, 화살표 `→ ▼ ──→`, 트리 `├─ └─`, 점선 `·····`
- 인라인 주석 `← 설명`, 차트 끝에 "핵심: ..." 한 줄 요약
- 프로세스 순서는 번호 `① ② ③`으로 표시

## 수정의 깊이 — 3단계를 항상 완주한다

> "동작하게 만드는 건 1단계일 뿐. 구조를 고치고, 문서로 남겨야 끝이다."

```
  ① 원인 분석    왜 이 버그가 구조적으로 가능했는가?
                  증상(플래그·분기 추가)이 아닌 근본 원인을 찾는다.
  ② 적절한 계층에서 수정
                  개별 프로젝트 문제 → 프로젝트 코드에서 수정
                  여러 프로젝트에서 반복될 패턴 → Core로 승격해서 Template화
                  임시 해결이 불가피하면 TODO + 구조적 해결 방향 명시
  ③ 문서화 + 재발 방지
                  프로젝트 CLAUDE.md에 패턴 기록 (흔한 실수, 구조적 결함 방지)
                  CHANGELOG 업데이트
                  새 분기 추가 시 테스트도 추가
```

**하지 않는 것:**
- 증상만 고치고 끝내기 (하드코딩 플래그, 특정 케이스 분기 추가)
- 프로젝트에서 반복되는 패턴을 매번 프로젝트마다 개별 구현
- 고쳤으면 됐지 하고 문서·테스트 생략

## 작업 방식
- 큰 변경은 계획 먼저 제시 후 진행
- 버그 수정 이력은 프로젝트 MEMORY.md에 기록
- 새 프로젝트·아키텍처 변경은 프로젝트 CLAUDE.md에 반영
- **같은 실수가 2번 이상 발생하면 프로젝트 CLAUDE.md "흔한 실수" 테이블에 추가한다.** 사용자가 교정한 패턴은 재발 >방지를 위해 반드시 문서화.

## 문서 계층 — 어디에 뭘 쓰는가

```
~/.claude/CLAUDE.md                  ← [이 파일] 글로벌. 모든 프로젝트에 적용.
                                       코딩 원칙, 설계 철학, 커뮤니케이션 스타일.

프로젝트/CLAUDE.md                   ← 프로젝트별. git에 포함 (팀 공유).
                                       아키텍처 규칙, 클래스 구조도, 절대 규칙.

~/.claude/projects/{경로}/           ← 프로젝트별. 로컬만 (본인만).
  memory/MEMORY.md                     실무 지식, 아키텍처 결정 이력, 버그 패턴.
```
